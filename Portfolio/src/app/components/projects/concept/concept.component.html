1 Hard-software requirements

The chat application has several hardware and software requirements:

  • Server to run the chat server on;
  • A computer to use the chat client on;
  • Internet connectivity from the computer to the server;
  • Server that runs the database (this can be the same as the one with the chat server).

2 Components

The components needed for the chat application are:

  • The client for the users;
  • The server that the clients connect with;
  • A database to storage data.

3 Layers

The components of the chat application server and client exists of three layers:

3.1 Client:

  • View layer, basically the design and the view of the client;
  • Logical layer, this does eventually needed logic;
  • Connection layer, this connects to the server.

3.2 Server:

  • Controller layer, this accepts incoming connections;
  • Logic layer, this handles logic such as user sessions, needed encryption;
  • Database logic, this layer connects and writes to a database.

4 Use cases

The use cases for the chat application are described below:

  • An user has the possibility to register an account;
  • An user has the possibility to login;
  • An user has the possibility to logout;
  • An user has the possibility to delete her/his account;
  • An user has the possibility to make a chat group;
  • The creator/admin of a group can manage its group;
  • An user has the possibility to send chats in a group;
  • An user has the possibility to send/accept friend invites to other players;
  • An user has the possibility to send messages to friends only.

5 Security

The needed security in the chat application is found both in the client and server:

5.1 Client:

  • Secure connection to the chat server with traffic encryption;
  • Input validation.

5.2 Server:

  • Secure connection to the client with traffic encryption;
  • Input validation;
  • Authentication (2FA);
  • Authorization;
  • Secure connection to the database;
  • Secure storage of data (hashed passwords, encrypted chats).

6 Data

The data that will be used in the chat application is as follows:

  • Account data, this contains usernames, passwords (hashed);
  • Two factor authentication data, the 2FA validate code;
  • Chats between users;
  • Database passwords;
  • Private keys for encrypting data (chats).

6.3 Multifactor authentication

One factor authentication

When a user will login, the system will ask for an username and password. This is the one factor authentication that will be used in the chat application as most basic authentication method.

Two factor authentication

When the user wants to have their account more private secured, there will be an option to enable two factor authentication with Google authentication. This will give the user an QR-code which can be scanned with the Google authenticator application on their phone. With this enabled, the user will get a 6 digit code which is unique and which get renewed every 60 seconds.

Three factor authentication

A third factor method will be some harder to enable. Biometric factors will be hard because not everyone has for example a fingerprint system on their device, this same goes for a camera or other ways. Also, when using a camera or another method, this has way more privacy risks for the application because there is an extra factor to save securely and it may be in conflict with article 9 of the AVG law. When this gets leaked or stolen, it has a higher privacy issue since this is human personally information which can be used for other systems as well.
