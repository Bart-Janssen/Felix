Concept chat applicatie Felix

Voor dit semester is het bij software security de bedoeling dat ik een applicatie maak waarin verschillende security aspecten ingebouwd zijn, zodat we deze applicatie kunnen gebruiken bij het Red team vs Blue team event. Tijdens de eerste workshop van Casper hebben we hier aandacht aan besteed. We hebben een concept opgezet voor een applicatie. Hierbij was het van belang om ook de autorisatie, rechten en privacy te bepalen over de data die binnen deze applicatie gebruikt wordt. Deze applicatie kan ook gebruikt worden om verschillende leerdoelen in aan te tonen. In dit document wordt ingegaan op wat voor applicatie ik ga maken en wat deze applicatie inhoudt.


1 Felix

Felix is de naam van de applicatie waarvoor dit concept is. Felix is een chatapplicatie waarin gebruikers vrienden kunnen toevoegen en met elkaar kunnen chatten. Het is ook mogelijk om een groep aan te maken en hierin mensen toe te voegen en daarin als groep te chatten. De gebruikers hebben een client ter beschikking die via de chatserver communiceert naar de andere gebruikers. De afbeelding hieronder laat zien hoe de verbinding loopt van de clients naar de server.

2 Functionaliteiten

Binnen Felix moeten gebruikers een account aan kunnen maken. Daarnaast hebben ze verschillende opties zoals het versturen van vriendschapverzoeken naar andere gebruikers. Deze gebruikers kunnen deze dan accepteren of wijgeren. Als ze deze geaccepteerd hebben dan kunnen ze met elkaar chatten. Ook kan iemand een groep aanmaken. De gebruiker die de groep aanmaakt is automatisch de eigenaar. Vervolgens kan deze mensen aan de groep toevoegen en iedereen die in de groep zit kan hier in chatten. Ook hebben de gebruikers de optie om two factor authentication in te schakelen voor een extra laag beveiliging op hun account. Hieronder een lijst met de functionele en non-functionele eisen waaraan de applicatie moet voldoen.

Functionele eisen

  • Een gebruiker moet een account kunnen registreren;
  • Een gebruiker moet kunnen inloggen;
  • Een gebruiker moet kunnen uitloggen;
  • Een gebruiker moet zijn/haar account kunnen verwijderen;
  • Een gebruiker moet een chat groep aan kunnen maken;
  • Een gebruiker moet een bericht in een groep kunnen versturen;
  • Een gebruiker moet vriendschapsverzoeken kunnen versturen, accepteren en weigeren;
  • Een gebruiker moet privé berichten kunnen versturen naar vrienden;
  • Een gebruiker moet two factor authenticatie in kunnen schakelen.

Non-functionele eisen

  • De wachtwoorden van de gebruikers moeten versleuteld opgeslagen zijn;
  • De server moet te allen tijde bereikbaar zijn;
  • De chats moeten versleuteld opgeslagen worden;
  • Het verkeer van en naar de server moet versleuteld zijn.

4 Software en Hardware

Felix heeft verschillende vereisten voor de software en hardware. De hardware voor Felix bestaat minimaal uit twee systemen, maar voor een correcte werking zijn drie systemen aangeraden (twee clients en een server). Hieronder een lijst met de vereiste voor de hardware en software:

Hardware

  • Server waar de chat server op draait;
  • Een computer voor de client;
  • Internet verbinding van computer naar de server;
  • Server voor de database (Dit kan dezelfde zijn als waar de chat server op draait).

Afbeelding 4 Design

Software

  • Ondersteuning voor Java;
  • Windows of Linux voor de server;
  • Windows of Linux voor de client.

5 Componenten

De componenten van Felix server en client bestaan uit drie lagen:

Client:

  • View laag, deze laag houdt het design volgens de wireframes;
  • Logical laag, deze laag handelt eventuele logische berekeningen af;
  • Connection laag, deze laag zorgt voor de connectie naar de server.

Server:

  • Controller laag, deze laag handelt aanvragen af;
  • Logical laag, deze laag zorgt voor de sessies en encryptie;
  • Database laag, deze laag zorgt voor connectie en encryptie naar de database.

6 Use cases

De use cases voor Felix staan hieronder benoemd:

  • Een gebruiker heeft de mogelijkheid om een account te registreren;
  • Een gebruiker heeft de mogelijkheid om in te loggen;
  • Een gebruiker heeft de mogelijkheid om uit te loggen;
  • Een gebruiker heeft de mogelijkheid om zijn/haar account te verwijderen;
  • Een gebruiker heeft de mogelijkheid om een groep aan te maken;
  • De administrator van een groep kan de groep beheren;
  • Een gebruiker heeft de mogelijkheid om berichten te versturen in een groep;
  • Een gebruiker heeft de mogelijkheid om vriendschapsverzoeken te versturen/accepteren;
  • Een gebruiker heeft de mogelijkheid om two factor authenticatie in te schakelen;
  • Een gebruiker heeft de mogelijkheid om two factor authenticatie uit te schakelen;
  • Een gebruiker heeft de mogelijkheid om privé berichten te versturen naar vrienden.

7 Security

Veel security is zowel in de server als client toegepast. De client doet checks op bepaalde security, dit zorgt voor minder aanvragen naar de server. De server doet deze check alsnog en deze is leidend. Hieronder een lijst van bepaalde security aspecten die in Felix voorkomen:

Client:

  • Data encryptie naar de server;
  • Input validatie.

Server:

  • Data encryptie naar de client;
  • Input validatie;
  • Authenticatie (2FA);
  • Autorisatie;
  • Data encryptie to the database;
  • Encrypte data (gehashde wachtwoorden, encrypted berichten).

8 Data

De data die in de database opgeslagen staat is als volgt:

  • Account data, dit bestaat uit gebruikersnamen en wachtwoorden (gehashed);
  • Two factor authenticatie data, de 2FA validatiecode;
  • Berichten van gebruikers.

Tabel 1 Autorisatie en rechten


Account data kan alleen beheerd worden door de eigen gebruiker, een systeem administrator moet accounts kunnen verwijderen.

2FA data kan niet eens ingezien door de gebruiker zelf.

Berichten zouden alleen zichtbaar moeten zijn door diegene van wie deze zijn. In een groep, de groepsbeheerder moet ongepaste berichten kunnen verwijderen.

Database wachtwoorden zouden alleen geüpdated moeten mogen worden door de systeem/server admins.

Private keys zouden alleen aangemaakt, geüpdated of verwijderd moeten mogen worden door de systeem/server admins.

Functionaliteiten zijn de functies van Felix die de gebruiker moet kunnen gebruiken.

Product inzien is wie de applicatie zelf (server/configuratie) moet kunnen inzien.

Product aanpassen kan alleen gedaan worden door de ontwikkelaars van de applicatie.

Tabel 2 CIAP


Account data is privé voor de gebruiker zelf. Dit is volledig persoonlijk en van bijvoorbeeld wachtwoorden bevatten, welke niet gewenst worden te delen.

2FA data is geheime data voor de 2FA instelling. Deze data is nodig als 2FA ingeschakeld is.

3FA data zou biometrische data kunnen bevatten, dit is hoog privé en mag door niemand anders ingezien worden.

Berichten zijn te allen tijde privé.

Database wachtwoorden zijn zeer belangrijk en de server is het enige systeem wat hier kennis van mag hebben.

Private keys worden gebruikt om data te versleutelen, niemand mag deze inzien.


Multifactor authenticatie

One factor authenticatie

Wanneer een gebruiker inlogt, vraagt het systeem om een gebruikersnaam en wachtwoord. Dit is de enige factor-authenticatie die in de chattoepassing als meest elementaire wordt gebruikt authenticatiemethode.

Two factor authentication

Wanneer de gebruiker zijn account meer privé wil beveiligen, is er een optie om tweefactorauthenticatie in te schakelen met Google-authenticatie. Dit zal geven de gebruiker een QR-code die kan worden gescand met de Google authenticator-applicatie op zijn telefoon. Als dit is ingeschakeld, krijgt de gebruiker een 6-cijferige code die uniek is en die elke 60 seconden worden vernieuwd.

Three factor authentication

Een derde factor methode zal wat moeilijker in te schakelen zijn. Biometrische factoren zullen moeilijk zijn omdat niet iedereen bijvoorbeeld een vingerafdruksysteem op zijn apparaat heeft staan hetzelfde geldt voor een camera of andere manieren. Ook bij het gebruik van een camera of een andere methode heeft dit veel meer privacyrisico's voor de applicatie omdat er een extra factor is veilig opslaan en het kan in strijd zijn met artikel 9 van de AVG-wet. Wanneer dit wordt uitgelekt of gestolen, heeft dit een groter privacyprobleem, aangezien dit persoonlijk menselijk is informatie die ook voor andere systemen kan worden gebruikt.
