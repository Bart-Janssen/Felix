Excercises en challenges

Ik wil de Canvas structuur deels aanhouden. Ik wil kijken welke standaard Canvas excercises en challenges ik leuk vind. Daarnaast wil ik nog verder eigen uitdagingen zoeken. Deze staan in mijn eigen MoSCoW structuur en prioriteit beschreven.


Eigen uitdagingen:

  • C: Mezelf verdiepen in het compileren van applicaties, hoe dit werkt en wat hierbij komt kijken;
  • S: Digitaal ondertekenen van licenties;
  • M: Reverse engineering en zo bijvoorbeeld malware/cracks kunnen toepassen;
  • S: Onderzoek doen naar quantum networking en encryptie op dat niveau;
  • M: Mezelf meer verdiepen in CPU instructies, RAM management, Assembly programmeertaal om zo meer kennis te krijgen hoe applicaties (inhoudelijk) werken, dit zal deels al naar voren komen in de reverse engineering challenges.

De canvas excercises:

  • M: Implement the google authenticator two-factor authentication mechanism on application and on server level;
  • M: Make a misuse case (or abuse case) design for an IT system, service or application that you are developing. Start with a global use case diagram of your IT system (or application). Research and add the best counter measures on application, system and network level to your design to prevent the misuse cases;
  • S: Fix some security bugs in an application or system environment. (For example the results from the first Red Team vs. Blue Team event). Report on the bugs and your fixes in your portfolio;
  • S: Perform the AES encryption operations manually on one block of 128-bits data with a 128-bit (round) key. Start with the data block in a 16 byte state and apply the four AES-operations on the state. Add a link to your portfolio page on cryptography to an appendix with the calculations;
  • M: Implement data encryption on application code level. Select a trusted cryptographic library for your programming language, find/select a use case for data encryption and implement the data encryption with the use of AES and the best fit block chaining method. Be aware of the difficulty of secure key handling (you don't want to hard code keys in your code) and find a good way to solve this problem. Explain your approach and the essential coding snippets and an example of execution of the code in your portfolio;
  • M: Learn the mathematical basics of RSA (and Diffie Hellman) public key cryptography : Modular arithmetic, the (extended) Euclidean algorithm, Euler's phi-function, Fermat's and Euler's theorem. Add a link to your portfolio page on cryptography to an appendix with some relevant calculations. Some example exercises;
  • M: Add security logging to your design, implementation and testing. Design: Add a list of security related events, the logging technology, code example, log message format. Implementation: Add a standard logging mechanism in your code, add the log events. Testing: test for the log events to be triggered, check if the logging is done correctly;
  • M: Add fuzzing tests for an application that you are developing, for example in the form of unit tests. Create fuzzing tests aimed at discovering buffer overflow issues and Denial of Service issues. If you are developing a web application you could also add fuzzing tests for XSS and/or SQL injection.

De canvas challenges:

  • M: Apply all security by design steps and techniques in your R&D project:
    • Gather and discuss security requirements;
    • Define security threats for the system to develop and perform a risk analysis;
    • Put the threats in a secure design through misuse cases;
    • Work out security controls to limit the threats in your misuse cases;
    • Implement the security controls;
    • Test the security controls by developing test cases for the threats. Add fuzzing tests;
  • M: Join the Red Team - Blue Team test event! Deploy your developed system (application(s), servers, tooling, network functionality) in an operational testing environment. Configure your system and application logs to be send to a central log server or SIEM server in the combined test environment. Ask the blue teaming students how they can receive your logs (technology, port number, secure channel);
  • S: Advise other developers (e.g. team members in your or another R&D project) how to apply security in their developments;
  • C: Implement the AES algorithm in your favourite programming language. Note: It is not advised to implement cryptographic algorithms yourself for operational use unless you are a professional cryptographer, but for studying the algorithm it is a very nice challenge. Start with implementing support for just the 128 bit key length (you could always add the other key lengths later). Start with one block of data, then add the blockchaining (see [5], paragraph 5.1) mode ECB (note: this is easiest but unsafe is most applications), use null-padding as a padding scheme. After implementing ECB, add CBC, then other block chaining modes (CTF, CFB, ...) if you like.
    Or:
    Implement the RSA algorithm in your favourite programming language. Start with encryption and decryption of a byte of data with integer length keys. You can manually calculate the needed public key pair. Then expand to for example 2048 bit keylengths with public keypairs generated by openssl. You can implement Optimal Asymmetric Encryption Padding (OAEP) for padding. You can also implement digital signatures. For this, use a private key and your decryption algorithm for signing a sha-2 hash of data, and the public key to validate the signature;
  • C: Is (your) implemented (AES or RSA software) protected against side channel attacks?
  • M: Prepare towards the Red Team - Blue Team exercise (day/event) together with the Blue and Red Teaming students. Set up your secure systems and applications in an operational test environment. Connect/send your applications and systems security logs to the central log servers or SIEM servers from the Blue Teaming students. Support the Blue Teaming students to be able to add security sensors to your operational test network. Use the experience of the first event to improve your secure designs and implementations.
